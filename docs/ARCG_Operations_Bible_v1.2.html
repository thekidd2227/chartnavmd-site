§1

Overview

Purpose, captured data, business outcome, target audience, and technical stack

1.1 Purpose

The ARCG Systems Assessment Form is an intake qualification tool deployed at arcgsystems.com/assessment. Its purpose is to collect structured operational data from prospective clients, route that data to [email protected], and trigger a custom workflow plan delivered within 24 hours. The form is the primary top-of-funnel conversion mechanism for the website.

1.2 What the Form Captures
→
Business category — drives Step 2 branch selection (8 options as of v1.1)
→
Industry-specific operational context — 1–3 questions, varies by branch
→
Universal qualification data — problem description, tools, users, volume, urgency, budget, LTV, desired outcome
→
Contact information — business name, contact name, work email
1.3 Business Outcome

On successful submission, ARCG receives a notification email at [email protected] (forwarded to [email protected]) containing all form fields. The operations team reviews the intake and delivers a custom workflow plan and fixed-fee proposal within 24 hours. No discovery call is required before delivery.

§8

Logic Risks / Edge Cases

Identified risks ordered by severity — includes new risk introduced by Healthcare branch

HIGH

Branch Variable Pollution

When a user selects GovCon, fills Step 2A, then goes Back and switches to "Service Business" (or any other type including Healthcare), the govcon fields retain their values and are submitted with the new branch's data.

Owner email contains irrelevant populated fields from prior branch selection. Data is noisy but not missing.
HIGH

Healthcare Branch Routing Failure Risk (NEW)

If Make PROD-01 routing logic is not updated to check for businessType === "Healthcare / Medical Billing", submissions from Branch 2H will incorrectly route to the Leads table instead of Audit Intake. This is a silent failure — no error is thrown.

Healthcare submissions appear in Leads table. Audit Intake stays empty. Owner misses specialized diagnostic context. PROD-01 blueprint must be verified after any schema change.
HIGH

No Confirmation Email

No automated confirmation is sent to workEmail after submission. User sees success screen but receives no email proof of submission.

Reduced trust. Users may resubmit thinking data was lost. No record in their inbox.
HIGH

No Failsafe / Local Storage

If Web3Forms fails (network error, API downtime), submission data is permanently lost. No retry mechanism, no localStorage backup, no error logging to any system.

Unrecoverable data loss on submission failure. User sees error banner but their intake is gone.
MEDIUM

Budget Disqualification Gap

Users selecting "Under $1K" are unqualified but receive the same success screen and follow-up process as qualified leads.

Wasted review time on leads that can never convert. No DQ logic exists.
MEDIUM

Thin Professional Services Branch

Branch 2E has only 1 question (professionalPipelineIssue). This is the weakest diagnostic branch — 1 field vs 3 in most others including the new Healthcare branch.

Insufficient context to build a meaningful workflow plan. Higher misdiagnosis probability.
MEDIUM

Step 3 Question Volume

Step 3 has 9 required dropdown fields — the longest single step. On mobile, each dropdown requires multiple taps. No progress indicator within the step.

Potential abandonment at Step 3. Most likely drop-off point in the form.
LOW

No CRM / Database Storage

All submission data exists only in the Web3Forms notification email. No structured database, Airtable, or admin panel stores submissions independently.

No pipeline visibility. No queryable submission history. Operational risk at scale.
